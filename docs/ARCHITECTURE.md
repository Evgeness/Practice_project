# Архитектура и логика работы

## 1. Общая схема

```text
Браузер
  │
  ▼
Nginx + React/JavaScript (front:80)
  │ /api/*
  ▼
FastAPI (app:8000)
  ├── PostgreSQL: документы и история запросов
  ├── Elasticsearch: полнотекстовый индекс чанков
  ├── Redis: кеш поисковой выдачи, TTL 300 секунд
  └── /metrics: Prometheus
          │
          ▼
       Grafana
```

Проект логически разделен на сервисы и запускается одним `docker compose up --build`.

## 2. Backend

### 2.1. API-слой

- `app/api/auth.py` — вход и проверка текущего пользователя.
- `app/api/documents.py` — загрузка и список документов.
- `app/api/search.py` — поиск и история запросов.
- `app/api/health.py` — состояние зависимостей.
- `app/api/dependencies.py` — Elasticsearch, Redis и проверка Bearer-токена.

API-слой не содержит сложной обработки текста. Он координирует сервисы, транзакции и HTTP-ответы.

### 2.2. Сервисный слой

- `document_parser.py` проверяет файл и извлекает текст.
- `chunking.py` нормализует пробелы и режет текст на части.
- `elasticsearch_service.py` создает индекс, индексирует чанки и выполняет поиск.
- `redis_service.py` кеширует сериализованный JSON.
- `auth_service.py` проверяет учётные данные и подписывает токены HMAC-SHA256.
- `metrics.py` объявляет счетчик и гистограмму Prometheus.

### 2.3. Слой данных

PostgreSQL хранит только метаданные, а не бинарные файлы.

`documents`:

- `id` — UUID;
- `file_name`, `file_type`, `file_size`;
- `status`: `processing`, `ready`, `error`;
- `chunks_count`;
- `error_message`;
- `uploaded_at`.

`search_history`:

- текст запроса;
- число результатов;
- время запроса.

Elasticsearch хранит один документ индекса на каждый чанк:

```json
{
  "document_id": "uuid",
  "chunk_id": "uuid:3",
  "file_name": "lecture.pdf",
  "page_number": 7,
  "text": "..."
}
```

## 3. Поток регистрации и авторизации

1. Для регистрации Frontend отправляет логин и пароль на `/api/v1/auth/register`.
2. Backend нормализует логин, проверяет уникальность и сохраняет пользователя в таблицу `users` PostgreSQL.
3. Пароль не хранится в открытом виде: применяется PBKDF2-SHA256 с отдельной случайной солью для каждого пользователя.
4. После регистрации либо успешного входа через `/api/v1/auth/login` создаётся HMAC-подписанный токен с именем пользователя и временем истечения.
5. Frontend сохраняет токен в `localStorage`.
6. Axios добавляет `Authorization: Bearer ...` ко всем защищённым запросам.
7. Загрузка, список документов, поиск и история отклоняют запросы без валидного токена.
8. Роли отсутствуют: все пользователи работают с общей базой знаний.

## 4. Поток загрузки

### Шаг 1. HTTP-запрос

Фронтенд отправляет `multipart/form-data` на `/api/v1/documents/upload`.

### Шаг 2. Валидация

Backend читает максимум `20 МБ + 1 байт`, поэтому слишком большой файл не загружается бесконечно в память. Проверяются:

- наличие имени;
- расширение `.pdf` или `.docx`;
- непустое содержимое;
- размер не более установленного лимита.

При ошибке возвращается `400 Bad Request` с полем `detail`.

### Шаг 3. Создание метаданных

Генерируется UUID. В PostgreSQL создается запись `processing`. Это позволяет видеть документ в списке даже во время обработки.

### Шаг 4. Извлечение текста

- PDF: `pdfplumber` обходит страницы и сохраняет их реальные номера.
- DOCX: `python-docx` извлекает абзацы. Точный номер визуальной страницы DOCX без движка верстки недоступен, поэтому используется страница `1`.

### Шаг 5. Чанкинг

Каждая страница разбивается отдельно. Размер — до 1000 символов, перекрытие — около 100 символов. Разрез по возможности переносится на пробел, чтобы не ломать слова.

Перекрытие нужно, чтобы фраза на границе двух чанков не потерялась для поиска.

### Шаг 6. Индексация

Чанки отправляются в Elasticsearch одной bulk-операцией. После индексации выполняется refresh, чтобы загруженный документ сразу находился в учебной демонстрации.

### Шаг 7. Финальный статус

- успех: `ready`, сохраняется число чанков;
- ошибка чтения: `error`, HTTP 400;
- ошибка инфраструктуры: `error`, HTTP 500.

## 5. Поток поиска

1. Запрос нормализуется: лишние пробелы удаляются.
2. Из запроса, страницы и размера строится SHA-256 ключ Redis.
3. При cache hit JSON возвращается без Elasticsearch.
4. При cache miss выполняется `multi_match` по `text`.
5. Русский анализатор выполняет lowercase, удаление стоп-слов и стемминг.
6. `fuzziness=AUTO` помогает находить небольшие опечатки.
7. Elasticsearch возвращает `_score` и фрагмент с маркерами `[[HIGHLIGHT]]`.
8. Результат кешируется на 300 секунд.
9. Запрос сохраняется в PostgreSQL.
10. Prometheus увеличивает счетчик hit/miss и измеряет длительность поиска.

## 6. Безопасная подсветка

Backend не возвращает HTML-теги из документа. Вместо `<mark>` используются текстовые маркеры:

```text
[[HIGHLIGHT]]поиск[[/HIGHLIGHT]]
```

React разбирает их на безопасные React-элементы. Это исключает вставку HTML из загруженного документа.

## 7. Frontend

- `UploadZone` обрабатывает drag-and-drop и множественную загрузку.
- `DocumentsList` показывает дату, размер, число чанков и статус.
- `SearchPanel` запускает поиск по Enter или кнопке.
- `SearchResults` отображает карточки, score, страницу и подсветку.
- пагинация переключает страницы по 10 результатов;
- CSS содержит адаптацию от 320 px до широких экранов.

Прогресс делится на две фазы: браузерная загрузка и серверная индексация. Backend отвечает только после полной индексации, поэтому после отправки 100% интерфейс показывает 90% и состояние «Индексация...», затем 100% и «Готово».

## 8. Запуск контейнеров

Порядок готовности задается healthcheck:

1. PostgreSQL, Elasticsearch и Redis;
2. backend;
3. frontend, Prometheus;
4. Grafana.

Backend при старте создает SQL-таблицы и индекс Elasticsearch. Повторный запуск не удаляет данные, потому что используются Docker volumes.

## 9. Отказоустойчивость прототипа

- недоступный Redis не ломает поиск: система работает без кеша;
- backend ждет PostgreSQL и Elasticsearch с повторными попытками;
- ошибки загрузки фиксируются в статусе документа;
- Docker healthcheck показывает деградацию зависимостей;
- `.env` не попадает в Git.

## 10. Ограничения

- OCR для сканированных PDF не реализован;
- DOCX не имеет точных номеров визуальных страниц;
- регистрация доступна, но отсутствуют восстановление пароля, подтверждение email и роли;
- бинарные файлы не сохраняются после индексации;
- удаление и переиндексация документов не входят в минимальный набор требований.
